## 0. Operating Rule (read first)

Effective this filing, TimeTribe stops minting one `CO-<N>-<slug>.md` file per change order. **Every past and future change order lives in this one file.**

**Why.** Auditing the 27-file corpus to build this ledger surfaced three failure modes that a single running file structurally prevents:
1. **A numbering collision** — two independently-ratified documents both titled themselves CO-26 (§4.1). **A second, same-shape collision (§4.8) surfaced hours later, the day of this same filing** — this ledger's own consolidation audit missed a live CO-27 document when resolving the first collision, inadvertently reassigning an already-claimed number. Proof the single-ledger convention below is necessary, not just tidy: even the audit that motivated it wasn't immune until every past CO lived in one place.
2. **Orphaned citations** — CO-18, CO-22, CO-23, CO-24 are referenced by number in other ratified documents but have no back-filled standalone source file in project knowledge (§4.2–4.4).
3. **A stale repo-mount snapshot** — the `/mnt/project` copy of CO-20 predates its own §0.4 audit-resolution amendment; the live, current version disagrees with the mounted one (§4.5).

**Going forward:**
- **New change order** → append a new dated entry under §2, next available number. No new file.
- **Amendment to a past CO** → a new `§0`-style amendment note *inside that CO's entry in this ledger*, same in-place-correction pattern CO-20/CO-21 already established. No ruling is silently deleted — only corrected or superseded with a note, same as always.
- **After every edit to this file**, log the change via the memory tool so the convention and the latest state survive across sessions without re-reading this file end to end.
- The historical `CO-16` through `CO-29` files remain on disk, unmodified, as the pre-consolidation audit trail. They are not deleted and are no longer the authoritative record going forward — this ledger is.

**CO-17 — Admission ⇄ Tier Separation**
*Status: RATIFIED (structure); floor **values** remain Growth/Founders sign-off · Historical file: `CO-17-admission-tier-separation.md` · Registers: D14 (reframed), D15, D16*

One word — "qualification" — was carrying two unrelated jobs: is this Studio admitted at all, and which tier does it land on. Splitting them fixed three defects: the 2–4 Seat gap, greenfield-studio lockout, and a retention-review misfire.

- **Decision 1 — Admission** (single platform bar, qualify via any one): `REVENUE` (trailing-90d ≥ Solo floor) · `APPOINTMENTS` (≥150/mo) · `REFERRAL` (member in good standing) · `SALES_VERIFIED_FOOTPRINT` (greenfield — signed lease + trade licence + installed Seats, verified via HQ Document Verification Queue).
- **Decision 2 — Tier fit** (post-admission only, highest tier whose qualifier is met, pure OR): Tribe (≥2 staffed Seats **or** Tribe revenue **or** Tribe appointments), Ecosystem (≥2 Locations **or** Eco revenue **or** Eco appointments), Solo (default).
- **D14 reframed:** the verified 2-Seat path is a Decision-2 tier-fit path only — never an admission shortcut.
- **D15:** admission = Solo-bar-or-referral-or-greenfield; tier = structural-OR-revenue-OR-appointments, symmetric (Tribe→Seats, Ecosystem→Locations). Solo floor tagged `is_admission_bar: true` only.
- **D16:** platform-retention review (HQ-AGM) keys off the **Solo bar** only, 2 consecutive cycles, revenue AND appointments — never a higher tier's floor. Structural-qualifier loss triggers a separate, softer **tier-fit review** instead.
- Marketing `/pricing`: **unchanged** — existence-only doctrine holds, zero mechanics exposed.
- Cascade: `Admission_Gate_values` → v2, all 7 `pricing-plan-*` §1/§1.1 re-authored (execution artifact: `pricing-cell-amendment-v1.md`), entitlements service, AppSync resolvers (`admissionQueue`, `decideAdmission`, `openTierFitReview`, `admissionStatus`).
